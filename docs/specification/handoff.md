# The Product → Builder Handoff

The handoff is the defining moment of the PDLC: the product person delivers an **Approved Product Baseline** to the builder person. The baseline is a versioned, git-revision-pinned, authority-tagged snapshot of the product definition — not a pointer at a moving working tree.

## Why a baseline

- **Independent velocities.** Product continues evolving the definition (next baseline) while the builder implements the approved one.
- **Authority clarity.** The builder knows exactly which artifacts are binding requirements, which are context, and which are advisory suggestions (e.g., prototype schemas).
- **Traceability.** Implementation artifacts, tests, and release notes trace back to a specific baseline, not to "the docs at some point."

## Baseline manifest

`pdlc-docs/baselines/PB-<id>/baseline.yaml`:

```yaml
apiVersion: pdlc.productbuildershq.org/v1
kind: ProductBaseline
metadata:
  id: PB-2026-Q3-014
  title: Multilingual Profile Settings
  approvedAt: 2026-07-19
  approvedBy: product-owner
  supersedes: PB-2026-Q2-009

sources:
  # authority: normative = binding requirement
  #            informative = context and rationale
  #            advisory = recommended approach; builder may deviate
  - path: docs/specs/source/prd.md
    revision: 7f2c3a1
    authority: normative
  - path: docs/specs/source/uxd.md
    revision: e4219bd
    authority: normative
  - path: docs/entities/
    revision: e4219bd
    authority: normative
  - path: docs/specs/gtm/narrative-6p.md
    revision: 993c82f
    authority: informative
  - path: docs/api/openapi.yaml
    revision: 2a77d06
    authority: advisory
  - path: docs/specs/requirements.yaml
    revision: 7f2c3a1
    authority: normative       # the requirement-ID registry
  - path: acceptance/
    revision: 2a77d06
    authority: normative       # executable acceptance criteria — the production
                               # implementation must pass these same journeys
  - path: prototype/
    revision: 2a77d06

                               # via uxd.md; the implementation is not
  - path: locales/
    revision: 2a77d06
    authority: normative       # required locales and terminology are binding

evaluations:                   # quality evidence at approval time
  readiness:                   # the roll-up report, pinned like any source
    path: quality/readiness.json

    result: pass
  summary:                     # convenience extract from the readiness report
    visionspec: { prd: pass, uxd: pass }

  methodology: aws-aidlc
```

## Handoff mechanics

1. **Freeze.** All stage gates pass; the baseline manifest pins every source to its git revision; approval is recorded in `pdlc-docs/audit.md`.
2. **Export.** VisionSpec export targets generate the builder-methodology inputs (AI-DLC Vision/Requirements documents, Spec Kit workspace seed, OpenSpec, GitHub Issues/Jira). Exports are projections — the baseline remains canonical.
3. **Context strategy.** Per methodology, the builder consumes context as `direct` (reads canonical files at pinned revisions), `compiled` (generated input package only), or `hybrid` (compiled entry point linking back to canonical sources). Hybrid is the default.
4. **Trace forward.** The builder workspace records the baseline ID (e.g., in `aidlc-docs/` state); implementation artifacts reference requirement IDs from the baseline's specs.
5. **Feed back.** Builder findings (feasibility, cost, constraints) are logged and become inputs to the next baseline revision — the definition is corrected upstream, never patched downstream.

## Authority rules for builders

- `normative` sources define **what must be true**. Deviations require a baseline revision, not a local decision.
- `informative` sources explain **why** — use them to resolve ambiguity in normative sources.
- `advisory` sources show **one validated way** — the prototype's Ent schema and ER model prove coherence; the builder owns the production design.
- On conflict, `normative` beats everything; among normative sources, the reconciled `spec.md` traceability matrix is the tiebreaker, and unresolved conflicts are a baseline defect to escalate.
