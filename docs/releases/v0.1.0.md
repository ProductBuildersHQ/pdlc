# PDLC v0.1.0

*Released 2026-07-20*

The first release of the ProductBuildersHQ **Product Development Lifecycle (PDLC)**: the normative specification for how a product person delivers a complete, evaluated product definition to a builder, together with a Go module and CLI that make conformance a checkable artifact rather than a matter of discipline.

## What's in this release

### The specification

The `specification/` directory defines the lifecycle end to end:

- **Lifecycle** — the stages (Discovery, Definition, Experience & Prototype, API, Documentation, Localization, Baseline & Handoff), their entry/exit gates, and the normative translation ordering (guides before UI strings).
- **Project layout** — the canonical filesystem contract every artifact resolves to, composing PDLC-managed directories with tool-native workspaces (`aidlc-docs/`, Spec Kit) that are never reorganized.
- **Review site** — how every artifact assembles into one static site for human review, served natively from `docs/` with projection reserved for what cannot move.
- **Quality** — continuous evaluation composing design-system, API-style, accessibility, localization, and consistency judges into a single readiness report.
- **Requirements & acceptance** — a stable requirement-ID registry and executable acceptance tests (browser-driven journeys) that transfer across the handoff.
- **Adoption** — the agent-driven procedure for migrating an existing repository to the layout under human approval.
- **Handoff** — the Product Baseline, the versioned, authority-tagged contract between product and builder.

### The tooling

A Go module (`github.com/ProductBuildersHQ/pdlc`) implementing the machine-checkable core:

- **Layout contract** as Go types with an embedded `layout.yaml` — canonical paths, per-profile requirement levels, authority, and detection heuristics.
- **Classifier** that inventories a repository against the contract, skipping immovable and ignored directories and flagging conflicts for human resolution. Content-only matches are low confidence and never auto-proposed as moves.
- **Readiness evaluator** producing a [structured-evaluation](https://github.com/plexusone/structured-evaluation) report: presence on the checklist scale, quality from tool leaf reports, and `TOOL_NOT_RUN` semantics so an unrun tool holds readiness at *conditional* rather than passing silently.
- **CLI** (`pdlc`) with `check`, `inventory`, `init`, and `layout` commands.
- **Go-first JSON Schema** generated from the types for the layout and project manifests.

## Try it

```bash
go build -o pdlc ./cmd/pdlc

# scaffold a project manifest
pdlc init --id my-product --profile standard

# see where every artifact belongs
pdlc layout show --profile full

# classify an existing repository against the contract
pdlc inventory

# evaluate readiness and report gaps
pdlc check
```

## Design notes

This release is deliberately scoped to the **evaluation keystone** — the layout contract and readiness checker — because it is what turns the specification from prose into something enforceable, and it works brownfield-first: pointed at an existing project, its gap report *is* the migration plan. It was pressure-tested against a real repository during development, which drove the classifier's ignore-list and confidence heuristics.
